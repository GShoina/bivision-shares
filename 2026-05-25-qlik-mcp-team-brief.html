10. Parquet — The Missing Data Layer

Owner-მა აღმოაჩინა, რომ პირველი ბრიფი Parquet-ის გარეშე incomplete-ი იყო. ეს არ არის nice-to-have — Parquet არის modern AI+MCP data exchange-ის standard format. სამი ფაქტობრივი findings რომელიც არსებითად ცვლის ბრიფის სცოპს:

🎯 Finding #1 — Qlik native Parquet (FACT, since Feb 2024)

Qlik Cloud + Enterprise on Windows: single-line STORE statement ქმნის Parquet ფაილს. ნულოვანი learning curve Qlik dev-ისთვის. 15 $parquet-* type-control tags. JSON-RPC integration. Multi-table single Parquet ("mart" pattern per qlikdork.com).

// Qlik load script — owner's healthcare client TODAY, zero new tools
STORE * FROM CashFlow
  INTO [lib://DataFiles/snapshots/cf_$(=Date(Today(),'YYYYMMDD')).parquet] (parquet);

შედეგი: dated Parquet snapshot ყოველი reload-ის შემდეგ → queryable Claude-ის მიერ DuckDB MCP-ით, Excel Power Query-ით, Python-ით — 5-15× პატარა ვიდრე Excel, type-fidelity preserved. Excel-ი ხდება არასაჭირო brittle handoff layer.

🏗️ Finding #2 — Qlik Open Lakehouse (FACT, GA 2025-09-16)

Qlik-მა შექმნა fully managed Apache Iceberg service Qlik Talend Cloud-ში, BYOC კლიენტის VPC-ში. Multi-engine access: Snowflake / Athena / Spark / Trino / Databricks.

MetricQlik-reported
Infrastructure cost−50%
Ingestion cost−80%
Query speed2.5–5× faster
December 2025+ Kafka / Kinesis streaming

Bivision-ისთვის ორი implication:

  • 50M+ ₾ prospect-ი ჰკითხავს "lakehouse გაქვს?" → პასუხი ახლა დიახ via Qlik
  • Qlik თვითონ ფსონს დებს open Parquet/Iceberg-ზე → Bivision-Qlik partnership future-aligned-ი რჩება

🦆 Finding #3 — DuckDB + Parquet + MCP = production "no-Qlik" stack

MotherDuck remote MCP — launched Dec 2025. Local DuckDB MCP — 5 წუთში install Claude Desktop-ში. DuckDB stream-ავს Parquet-ს filter + projection pushdown-ით → Claude query-ობს 100MB+ datasets SQL-ით, raw bytes context-ში არ შემოდის.

11. როგორ მუშაობს MCP + Parquet — სიმართლე

წინა Parquet სექციამ მოგვცა შთაბეჭდილება, რომ Parquet არის "Qlik → Parquet → Claude" path-ი. ეს არ შეესაბამება სიმართლეს Qlik MCP workflow-ში. ვასწორებთ ახლა.

🎯 MCP-ის რეალური ბუნება — Tool Calling, NOT Bulk Transfer

MCP არის tool-calling interface. რეალური workflow:

  1. Claude ვაგზავნი tool call: qlik_create_hypercube(app_id, dimensions, measures, max_rows=5000)
  2. Qlik computes-ს ცხრილს server-side
  3. JSON აბრუნდება Claude-ის context window-ში

Hard limits:

  • Bintocher MCP: engine_create_hypercube rejects max_rows > 5000 OR columns × max_rows > 9900 with structured error + set-analysis/top-N hint
  • Qlik Engine: error 7009 calc-pages-too-large for >10,000 cells per page

ეს არ არის bug — ეს არის design. MCP გათვლილია "ერთი კონცენტრირებული პასუხი per call".

✅ რა შესაძლებელია · ❌ რა შეუძლებელია MCP-ით

📐 Parquet-ის რეალური ადგილი — Infrastructure Layer, NOT Transport

Parquet არის Qlik-ის ქვემოთ, არა Qlik-სა და Claude-ს შორის MCP workflow-ში. Claude-ი ვერ "ხსნის" Parquet-ს Qlik MCP-ით — Claude აკეთებს API calls Qlik-თან. Parquet binary format-ია, არ ჯდება LLM context-ში.

რეალური Qlik MCP architecture:

[Source Systems (Fina/Oris/1C/Balance/Bank APIs)]
         ↓ ETL pipeline
[Parquet files in S3/Azure Blob/local lake]
         ↓ Qlik Data Gateway / Direct Query / Load Script
[Qlik Cloud App — semantic model + master measures]
         ↓ MCP (tool calls, NOT bulk transfer)
[Claude / LLM client]

SKU #4 "Bivision Lite" სხვა path-ია: Claude → DuckDB MCP → SQL on Parquet. Parquet ARI ხელმისაწვდომი Claude-ისთვის DuckDB layer-ის გავლით — Qlik-license-less ICP-სთვის. ვალიდურია, მაგრამ ნუ აღვრევთ Qlik MCP path-თან.

MCP magic NOT in dashboard building — that's a one-time setup. MCP magic IS in continuous interactive analysis. CFO + Claude უხდება ცოცხალი Q&A. ეს არის Bivision Lite + Qlik MCP-ის ერთიანი value proposition: კლიენტს ვაძლევთ "Qlik analyst" 24/7-ზე, ერთეული საათიანი setup investment-ით.
