⚙️ Deploy Approach — Selected

✓ SELECTED: WP Page Template
  • Create template-v10.php in active theme
  • Assign to new WP page, set as homepage
  • WordPress still handles /r/*, /wp-login, all routing
  • Rollback: change homepage setting (30s, no server access needed)
  • No Apache/LiteSpeed config changes
✗ Apache Rewrite Rule
  • Edit httpd.conf via WinRM
  • Risk: LiteSpeed may override Apache config
  • Risk: breaks /r/* routing if not ordered correctly
  • Rollback needs WinRM + Apache restart
  • More moving parts

Why NOT Apache rewrite

Server uses LiteSpeed Cache plugin — this suggests LiteSpeed Web Server or at minimum LS cache integration. Apache httpd.conf edits may be ignored or override in unexpected ways. WP page template approach is lower risk, fully reversible without server access.

Phase 1 — Staging on production server (zero downtime)
  1. Via WinRM: write template-v10.php to active theme directory (chunk-write, .NET ZipFile if needed)
  2. Create new blank WordPress page "BiHub v10" — assign template-v10
  3. Access bihub.ge/?page_id=[new-page-id] — verify v10 renders
  4. Test: registration modal → bihub.ge/wp-login.php link works
  5. Test: card click → bihub.ge/r/[slug] works (new tab)
  6. Test: notify strip form submit → /notify.php returns 200
  7. Screenshot: compare with GitHub Pages preview
Current homepage still live. New page accessible by direct URL only.
Phase 2 — Homepage swap (~30s visible change)
  1. WP Admin → Settings → Reading → "A static page" → set Front Page to "BiHub v10"
  2. LiteSpeed Cache → Purge All
  3. Immediately: curl -I https://bihub.ge — verify 200 + v10 content
  4. Mobile smoke test (Playwright, 375px viewport)
  5. Desktop smoke test (1440px viewport)
If smoke test fails → immediately revert Reading settings → back to old homepage in 30s
Phase 3 — GitHub deploy automation (optional, post-launch)
  1. Add WINRM_HOST + WINRM_USER + WINRM_PASS to GitHub Secrets
  2. Create .github/workflows/deploy.yml: on push to production branch → WinRM deploy v10.html updates
  3. Test: push change → verify auto-deploy
Not required for go-live. Enables future push-to-deploy workflow. Credentials in GitHub Secrets = encrypted, never in code.
Rollback Plan (any phase)
  1. Phase 1 rollback: delete new WP page — zero impact
  2. Phase 2 rollback: WP Admin → Settings → Reading → set Front Page back to original — 30 seconds
  3. No DNS changes needed — server stays same throughout
  4. No backup required — original WP home untouched

⚔️ Viktor Challenge

Independent audit — assumptions challenged, gaps surfaced
CONFIRMED RISK [HIGH]
WP page template = still WordPress overhead. Every request to the homepage will boot WordPress, load all active plugins, query the database — just to serve a static HTML file. If bihub.ge has heavy plugins (WooCommerce, ACF, security plugins), homepage TTFB will be 300-800ms vs <50ms for a true static file. The "WP template" approach trades simplicity for performance. Acceptable for MVP but not for scale.
CONFIRMED RISK [HIGH]
template-v10.php nested HTML structure. A WordPress page template that outputs a full HTML document (v10 has its own <!DOCTYPE>, <head>, <body>) will result in invalid HTML: two <html> roots if WP's get_header() is called, OR completely bypassed WP head if it's not — meaning WP plugins that hook into wp_head() (analytics, SEO, security nonces) won't run. Both outcomes have consequences. This requires explicit architectural decision: either (a) use WP as pure router and output raw HTML bypassing all WP head, or (b) strip v10's <head> and let WP own it. Neither is in the plan.
CONFIRMED RISK [HIGH] — GAP
/notify.php — what does it do exactly? The plan says "deploy /notify.php" but there is no notify.php written yet. Who writes it? What does it save leads to? Database? Email? Third party? This is a lead capture endpoint that does not exist. "Deploy" implies a file exists somewhere to deploy. It does not. This is a blocker for Phase 0, not a Phase 3 item.
UNVERIFIED ASSUMPTION [MED]
"No DNS changes needed." This assumes bihub.ge's DNS A record points to 185.229.111.201 and the plan keeps it there. Verified? If the domain is behind Cloudflare proxy (orange cloud), the actual server IP is masked. Cloudflare could have caching rules that serve a cached version of the old homepage for hours after the WP swap. Plan doesn't account for Cloudflare cache purge.
UNVERIFIED ASSUMPTION [MED]
Active WP theme is known and editable. Plan says "write template-v10.php to active theme directory." Which theme is active? If it's a purchased theme with auto-updates enabled, a theme update could delete template-v10.php. If it's a child theme, inheritance may add unexpected styles. This is unverified.
GAP [MED]
Registration modal links — post-migration path check. v10 modal hardcodes https://bihub.ge/wp-login.php. After migration, this should still work — but what about the registration confirmation email? Where does it redirect after WP registration? Currently it may redirect to the old WordPress homepage. Post-registration redirect = old WP page, not v10. User lands on wrong page. Not in plan.
GAP [LOW]
GitHub Actions deploy = credentials risk surface. Adding WinRM credentials to GitHub Secrets connects the GitHub repo (public or private) to full server access. If the repo is public, a misconfigured workflow could expose the secrets. Recommend: create a dedicated deploy user with write-only access to C:\wamp\www\wp-content\themes\[theme]\ only, not full WinRM admin.
⚖️ Plan is deployable IF: (1) notify.php is written first, (2) WP template HTML nesting is resolved explicitly, (3) Cloudflare cache is purged after swap. Everything else is manageable risk.
