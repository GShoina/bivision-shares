⚠️ Challenges / different opinions (3)

Challenge A (Rule 1 — iteration cap):
"Max 2/day" needs iteration definition. Proposal: iteration = material content change (new section, new recommendation, structural rewrite). Patch ≠ iteration (typo fix, deadline strip, ban-list removal per Apr 22 owner directive).
Counter-example today: Positioning v2 had owner-flagged issues (fake ERP metaphor, Qlik-in-thesis, Q6 pillar 4 Qlik emphasis, vendor-fiction). 4 patch edits = 1 iteration (fixing v2), not 4 iterations. v3 = new direction = counts as new.
Adopt: track v-numbers on material changes only. Patches in-place to current version.
Challenge B (Rule 6 — "hash" definition):
"path + line + hash" — hash = git commit hash (post-commit)? File content hash (SHA256)? Line-level content hash?
Proposal:
  • Post-commit: path:line + git hash (commit short SHA)
  • Pre-commit (working tree): path:line + grep output line (content proof)
  • Status-claim without either = rubber-stamp, banned
Challenge C (Rule 7 — Playwright debug ban):
Curl + grep ≠ sufficient when task is browser behavior itself: mobile viewport rendering, JS execution state, form-submission flow, actual font/layout render. Geo-Metri mobile audit Apr 20 blocked on Playwright disconnect for this reason.
Proposal refinement:
  • Default: curl + grep for code/content debug
  • Allowed: Playwright when verification requires actual browser state (viewport, JS runtime, form flow)
  • Ship-gate: always Playwright for final visual confirmation (Rule 7 core intent)
